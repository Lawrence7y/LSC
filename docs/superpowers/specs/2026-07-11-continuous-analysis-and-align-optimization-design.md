# 持续分析与一键对齐优化设计

> **目标读者：** 后续实施的开发者。本文只定义要优化什么、为什么优化、以及优化边界，不直接给出具体代码实现。

**Goal:** 将无畏契约持续分析与一键对齐链路优化为更稳定、更准确、可回退的闭环，让结果更接近“按回合产出高光”的实际使用预期。

**Architecture:** 保留现有前端工作台、后端分析器和持续分析消息链路，不做大规模重构。优先加固 `round_detector` 的回合裁剪与回退逻辑，其次收紧持续分析状态流与入队逻辑，最后统一前端状态展示与错误提示。

**Tech Stack:** TypeScript, React, Ant Design, Python, FFmpeg, OCR, WebSocket, pytest

---

## 背景

当前系统已经具备以下能力：

- 多房间一键对齐
- 无畏契约回合检测快速路径
- 买枪期裁剪、尾部裁剪、OCR 精修
- 持续分析状态展示
- 切片队列与导出进度展示

但实际使用中仍可能出现以下问题：

- 回合边界偶发偏早或偏晚
- 买枪期/准备阶段裁剪不稳定
- 尾部垃圾时间仍可能保留过多
- 持续分析与自动入队/导出之间的状态衔接不够清晰
- 用户难以判断“已经对齐好了，是否可以安全开始分析”

本次优化的目标不是增加新功能，而是把现有链路做得更稳、更准、更少人工判断。

## 目标

### 1. 提升无畏契约回合切分准确率

优化 `lsc/analyzer/round_detector.py` 的回合边界、买枪期裁剪和尾部裁剪，使输出更接近“纯战斗阶段片段”。

### 2. 强化持续分析的稳定性

让持续分析在增量分析、异常回退、状态更新和结果入队时行为一致、可预测，不出现“看起来在跑、结果却不可靠”的情况。

### 3. 收紧一键对齐与分析的联动

让多房间对齐结果成为持续分析与同步导出的可靠前置条件，并在前端清晰提示是否满足启动条件。

### 4. 改善错误提示与可观测性

失败时明确说明是音频、OCR、录制文件、对齐状态还是导出队列的问题，减少用户猜测。

## 非目标

- 不重做分析架构
- 不引入新的外部分析依赖
- 不把持续分析改成完全独立的新页面
- 不替换现有 WebSocket 消息体系
- 不在本次范围内重构全部前端布局

## 方案边界

本次优化只覆盖以下链路：

1. 一键对齐
2. 无畏契约持续分析
3. 高光回合裁剪
4. 自动入切片列表
5. 多房间同步导出前置校验
6. 失败回退与提示

## 设计方案

### A. 后端回合检测加固

重点文件：

- `lsc/analyzer/round_detector.py`
- `lsc/analyzer/pipeline.py`

设计要点：

1. 保留 `valorant` 快速路径，继续作为持续分析的主入口。
2. 让“音频 RMS → 回合边界 → 买枪期裁剪 → 尾部裁剪 → OCR 精修”的顺序保持稳定。
3. 把 OCR 视为增强，不作为硬依赖。
4. 在 OCR 失败、音频失败、样本不足时，明确回退策略：
   - OCR 成功时优先使用 OCR 边界
   - OCR 失败时回退纯音频结果
   - 音频不足时返回空结果并提供诊断信息
5. 对输出结果补充更明确的来源信息，便于前端和日志判断是否是“精修结果”。

### B. 持续分析链路加固

重点文件：

- `python-backend/main.py`
- `python-backend/server.py`
- `python-backend/message_bridge.py`
- `lsc/core/services/timeline_service.py`
- `lsc/core/services/ingest_registry.py`
- 其他持续分析相关处理服务

设计要点：

1. 启动持续分析前，强制校验：
   - 主房间存在
   - 目标房间存在
   - 目标房间有录制文件
   - 多房间对齐状态有效
2. 持续分析状态应明确区分：
   - `running`
   - `finalizing`
   - `completed`
   - `error`
3. 增量结果入队时避免重复添加同一片段。
4. 当分析失败时，保留失败原因，避免前端只收到“失败”而不知道原因。
5. 如果后端无法自动完成完整导出闭环，也要把状态标出来，不让前端误以为已完成。

### C. 前端工作台状态整理

重点文件：

- `lsc-electron/src/pages/Workbench/index.tsx`
- `lsc-electron/src/components/AnalysisProgress.tsx`
- `lsc-electron/src/types/index.ts`

设计要点：

1. 维持现有“分析导出”弹窗，但让按钮状态更明确。
2. 在按钮文案和提示中区分：
   - 一键对齐未完成
   - 录制文件缺失
   - 目标房间不一致
   - 持续分析正在运行
   - 持续分析失败回退
3. 持续分析状态条展示应包含：
   - 当前主房间
   - 当前阶段
   - 已分析时长
   - 高光数量
   - 更新时间或失败原因
4. 让用户更容易判断：什么时候可以开始分析，什么时候必须先对齐。

## 数据流

### 1. 一键对齐

用户在工作台点击一键对齐后：

1. 前端读取当前选中的房间。
2. 调用音频对齐流程。
3. 后端或前端对齐结果写入房间状态中的 `content_offset` 和 `align_group_id`。
4. 前端更新对齐后的可用状态。

### 2. 启动持续分析

用户选择主直播间并点击开始后：

1. 前端先检查目标房间和录制文件。
2. 前端向后端发送持续分析启动请求。
3. 后端进入 `running`。
4. 后端调用 Valorant 快速路径进行回合检测。
5. 检测结果写入持续分析状态和切片队列。

### 3. 回退与完成

若 OCR 不可用或样本不足：

1. 后端使用音频结果或返回空结果。
2. 后端记录失败原因。
3. 前端显示明确提示。
4. 如果任务完成，状态转为 `completed`；如果失败，状态转为 `error`。

## 错误处理

### 音频失败

- 返回空结果或明确失败状态
- 前端显示“音频捕获不足 / 音频检测失败 / 已回退到缓冲区对齐”等信息

### OCR 失败

- 不阻断主流程
- 继续使用纯音频结果
- 仅在日志和提示中记录 OCR 不可用

### 录制文件缺失

- 直接阻止分析启动
- 提示用户先录制或确认输出路径

### 对齐状态不一致

- 多房间分析前阻止启动
- 提示必须先一键对齐且对齐组一致

### 重复入队

- 后端/前端都应避免同一片段被重复加入切片列表
- 依赖唯一标识或时间窗口去重

## 测试策略

### 后端测试

1. 验证 `detect_valorant_rounds()` 在典型 Valorant 样本上能输出回合片段。
2. 验证 OCR 可用与不可用时的回退行为一致。
3. 验证买枪期裁剪不会把有效战斗段裁没。
4. 验证尾部裁剪不会过度截断。
5. 验证持续分析状态从 `running` 到 `completed` / `error` 的流转。

### 前端测试

1. 验证按钮在缺少录制文件时禁用或提示正确。
2. 验证分析状态卡片在不同阶段能正确显示。
3. 验证多房间对齐后才允许进入同步分析。
4. 验证持续分析失败时会显示原因。

## 实施顺序

### 第 1 阶段：后端回合检测加固

先改 `round_detector` 和相关分析编排，确保结果更准。

### 第 2 阶段：持续分析链路加固

再收紧启动条件、状态流和入队逻辑，确保结果更稳。

### 第 3 阶段：前端状态与提示整理

最后统一工作台按钮、状态条和失败提示，确保更好用。

## 验收标准

- 无畏契约回合切分结果明显更接近战斗期
- 买枪期和尾部垃圾时间减少
- 持续分析失败时有明确原因
- 多房间分析前置条件更清晰
- 用户能看懂当前处于哪个阶段
- 一键对齐后继续分析的成功率和可理解性提升

## 需要进一步确认的实现约束

- 是否允许在持续分析失败时自动切回纯音频结果
- 是否要求自动导出必须在前端完全无感知地完成
- 是否需要把“对齐完成”状态单独显示成常驻状态条
