# LSC 直播切片多人系统 - 全面技术审计报告

## 一、项目概述

**项目名称**: LSC (LiveStreamClipper) - 直播切片多人系统
**技术栈**: Python 3.10+ / PySide6 / FFmpeg / libmpv
**项目规模**: 约 15,000+ 行 Python 代码
**核心定位**: 多直播间同步录制（最多 12 路）、跨房间同步预览、一键导出精彩片段

### 2.2 核心设计模式

| 模式 | 应用位置 | 实现方式 |
|------|----------|----------|
| **Protocol + Registry** | 平台适配器 | `PlatformAdapter` Protocol + `_DEFAULT_ADAPTERS` 单例注册 |
| **Command** | 撤销/重做 | `UndoStack` + `Command` dataclass |
| **MVVM** | 录制页 | `RecordPage` (View) + `RecordingController` (ViewModel) |
| **Observer** | 主题系统 | `theme_changed` Signal + `connect_theme_changed()` |
| **Factory** | 多房间管理 | `controller_factory` / `preview_factory` 可注入 |
| **Strategy** | 编码配置 | `ExportProfile` 封装编码策略 |
| **异步 Worker** | 后台任务 | `QThread` + Signal 回调 |
| **三级降级** | FFmpeg 进程 | stdin 'q' → terminate → kill |
| **原子持久化** | 房间配置 | tmp + rename + bak |

## 三、模块详细分析

### 3.1 平台适配层 (`lsc/platforms/`)

**支持平台**: 抖音、B站、虎牙、快手、斗鱼、小红书、直链、通用回退

**架构亮点**:
- Protocol 定义统一接口，适配器必须无状态
- Host-based URL 路由表实现 O(1) 匹配
- TTL 缓存（成功 30s，失败 10s）防止重复请求
- 新增平台零侵入：实现 Protocol → 注册到元组

**关键文件**:
- `lsc/platforms/base.py:213-235` - PlatformAdapter Protocol
- `lsc/platforms/registry.py:35-44` - 适配器注册
- `lsc/platforms/registry.py:48-64` - URL 路由表

### 3.2 录制层 (`lsc/recorder/`)

**核心类**: `StreamCapture` (FFmpeg 封装) + `RecordingSession` (会话 API)

**关键特性**:
- 四级降级停止：stdin 'q' → wait(5s) → terminate → kill
- 共享 stderr 线程池（max_workers=2）节省内存
- 启动探活：5s 内确认 FFmpeg 写入数据
- 文件卡住检测：连续 3 次未增长判定为 stall

**关键文件**:
- `lsc/recorder/capture.py:198-300` - 启动录制
- `lsc/recorder/capture.py:302-382` - 停止录制（四级降级）
- `lsc/recorder/capture.py:444-477` - 健康检查

### 3.3 导出层 (`lsc/exporter/`)

**核心类**: `ClipExporter`

**关键特性**:
- copy 模式自动回退：start_sec > 0 或有滤镜时回退到 libx264
- 硬件编码器支持：NVENC/QSV/AMF 自动映射 preset
- 进度回调：通过 `-progress pipe:1` 实时解析
- 异步缩略图：ThreadPoolExecutor(max_workers=4) 后台生成

**关键文件**:
- `lsc/exporter/clip.py` - 切片导出核心逻辑

### 3.4 GUI 层 (`lsc/gui/`)

#### 3.4.1 主题系统 (`theme.py`)

**设计令牌架构**:
- `ThemeColors` frozen dataclass 作为单一真相源
- 20+ 颜色变量覆盖所有 UI 元素
- DARK/LIGHT 双主题预定义
- 1117 行 QSS 样式表通过 f-string 模板生成

**切换机制**:
1. `set_dark()` / `toggle_theme()` 修改全局状态
2. `_refresh_app_style()` 应用 QPalette + QSS
3. 遍历所有 widget unpolish/polish/update
4. 发射 `theme_changed` 信号通知订阅者

**关键文件**:
- `lsc/gui/theme.py:11-32` - ThemeColors 定义
- `lsc/gui/theme.py:35-81` - DARK/LIGHT 配色
- `lsc/gui/theme.py:211-1117` - QSS 生成

## 四、数据流转分析

### 4.1 录制流程数据流

```
用户输入 URL
    ↓
platforms/registry.py:parse_stream()
    ↓ (HTTP 请求 + 解析)
StreamInfo (platform, stream_url, headers, quality_urls)
    ↓
registry.py:select_quality()
    ↓
(Stream_url, selected_quality)
    ↓
recorder/capture.py:StreamCapture.start()
    ↓ (构建 FFmpeg 命令)
FFmpeg 子进程
    ↓ (写入 .mp4 文件)
capture.py:check_health() (每秒监控)
    ↓
CaptureResult (success, output_path, duration, file_size)
```

### 4.2 预览流程数据流

```
StreamInfo.stream_url
    ↓
MpvWidget.set_stream_url()
    ↓ (libmpv 播放)
视频渲染到 QOpenGLWidget
    ↓
ControlBar (播放/暂停/seek)
    ↓
InlineTimeline (选区标记)
```

### 4.3 导出流程数据流

```
用户设置 mark_in / mark_out
    ↓
ExportConfirmDialog (编码参数选择)
    ↓
ExportProfile (codec, crf, preset, filters)
    ↓
exporter/clip.py:ClipExporter.export()
    ↓ (FFmpeg 切片命令)
进度回调 → QProgressBar
    ↓
输出 .mp4 文件 + 缩略图
```

### 4.4 主题切换数据流

```
用户点击主题切换
    ↓
theme.set_dark(True/False)
    ↓
_refresh_app_style()
    ├── _apply_palette() (QPalette)
    ├── generate_stylesheet() (QSS 缓存)
    ├── app.setStyleSheet()
    ├── 遍历 widget unpolish/polish/update
    └── theme_changed 信号
        ↓
所有订阅者 _refresh_theme()
```

## 七、潜在问题与改进空间

### 7.1 架构层面

#### 问题 1: 巨型文件过多
- `multi_room.py` (~2000 行)、`record.py` (~1500 行)、`room_card.py` (987 行)
- **影响**: 可维护性差，合并冲突风险高
- **建议**: 拆分为更小的组件，遵循单一职责原则

#### 问题 2: 主题系统硬编码
- `theme.py` 中 1117 行 QSS 使用 f-string 模板
- **影响**: 修改样式需要理解整个模板系统
- **建议**: 考虑使用 Qt Quick/QML 或外部样式文件

#### 问题 3: 缺少状态管理框架
- 房间状态分散在 `RoomSession`、`MultiRoomManager`、各 UI 组件中
- **影响**: 状态同步复杂，容易出现不一致
- **建议**: 引入轻量级状态管理（如 Redux-like 模式）

### 7.2 性能层面

#### 问题 4: 全局心跳频率过高
- `QTimer(1s)` 驱动所有房间状态更新
- **影响**: 12 个房间时每秒执行大量操作
- **建议**: 按需更新，仅在状态变化时刷新 UI

#### 问题 5: QSS 重复生成
- 虽然有缓存，但主题切换时仍需遍历所有 widget
- **影响**: 切换主题时可能有短暂卡顿
- **建议**: 增量更新，仅更新变化的组件

#### 问题 6: 文件大小查询阻塞
- `SizeUpdateRunnable` 在 QThreadPool 执行，但仍可能阻塞
- **影响**: 大量房间时查询延迟
- **建议**: 使用异步 I/O 或降低查询频率

### 7.3 健壮性层面

#### 问题 7: 错误恢复不完整
- 自动重连仅 3 次，之后需要手动干预
- **影响**: 用户体验不佳
- **建议**: 增加重连策略配置，支持无限重连模式

#### 问题 8: 缺少录制完整性校验
- 录制完成后未验证文件完整性
- **影响**: 可能产生损坏的视频文件
- **建议**: 录制完成后运行 ffprobe 校验

#### 问题 9: 并发安全未完全保证
- 适配器要求无状态，但未强制执行
- **影响**: 潜在的并发 bug
- **建议**: 添加运行时检查或类型检查

### 7.4 用户体验层面

#### 问题 10: 缺少进度反馈
- 连接、录制、导出过程缺乏详细进度
- **影响**: 用户不知道当前状态
- **建议**: 添加进度条、状态提示、预计剩余时间

#### 问题 11: 错误信息不够友好
- 部分底层错误直接暴露给用户
- **影响**: 用户难以理解
- **建议**: 完善 `error_messages.py`，覆盖更多场景

#### 问题 12: 缺少快捷键
- 仅支持 Ctrl+1-4 和 Ctrl+T
- **影响**: 高频操作效率低
- **建议**: 添加更多快捷键（录制、导出、全屏等）

### 7.5 代码质量层面

#### 问题 13: 类型注解不完整
- 部分函数缺少类型注解
- **影响**: IDE 支持差，难以发现类型错误
- **建议**: 完善类型注解，启用 mypy 检查

#### 问题 14: 文档不足
- 缺少 API 文档和架构文档
- **影响**: 新开发者上手困难
- **建议**: 补充 docstring 和架构文档

#### 问题 15: 测试覆盖不均
- 平台适配器测试完善，但 GUI 测试较少
- **影响**: UI 回归风险高
- **建议**: 增加 GUI 组件单元测试
